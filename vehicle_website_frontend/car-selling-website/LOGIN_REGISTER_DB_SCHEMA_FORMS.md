# Login & Register Forms + Database Schema (FastAPI/SQLAlchemy)

This document maps the **Login** and **Register** UI input fields in `car-selling-website` to **backend request schemas** and a **database (DB) schema**.

## 1) Frontend fields (from React pages)

### Login form (`src/assets/pages/Login.jsx`)
Inputs:
- `email` (type=email, required)
- `password` (type=password, required)

### Register form (`src/assets/pages/Register.jsx`)
Inputs:
- `firstName` (required)
- `secondName` (required)
- `dateOfBirth` (type=date, required)
- `address` (required)
- `gender` (select: `male | female | other`, required)
- `email` (type=email, required)
- `password` (type=password, required)

> Note: The React forms currently `preventDefault()` (demo behavior). This doc focuses on the DB + backend schema alignment.

---

## 2) Recommended database model: `users`

### Table: `users`
Suggested columns:
- `id` (UUID primary key)
- `first_name` (string)
- `second_name` (string)
- `date_of_birth` (date)
- `address` (string)
- `gender` (enum: `male`, `female`, `other`)
- `email` (unique string)
- `password_hash` (string)
- `created_at` (timestamp)
- `updated_at` (timestamp)

#### Field → frontend mapping
- `firstName` → `first_name`
- `secondName` → `second_name`
- `dateOfBirth` → `date_of_birth`
- `address` → `address`
- `gender` → `gender`
- `email` → `email`
- `password` → `password_hash` (hash before saving)

#### Index/constraints
- Unique index on `email` (prevents duplicate accounts)
- `gender` limited to enum values

## 3) Backend request/response schemas (Pydantic)

### 3.1 Register endpoint request: `POST /api/auth/register`
Request JSON:
```json
{
  "firstName": "string",
  "secondName": "string",
  "dateOfBirth": "YYYY-MM-DD",
  "address": "string",
  "gender": "male|female|other",
  "email": "string",
  "password": "string"
}
```

Validation rules:
- `firstName`, `secondName`, `address`, `email`: non-empty
- `dateOfBirth`: valid date (`YYYY-MM-DD`)
- `gender`: must be one of allowed values
- `password`: non-empty (optionally enforce min length)

### 3.2 Register endpoint success response
Example:
```json
{
  "message": "Account created",
  "userId": "uuid-string"
}
```

### 3.3 Login endpoint request: `POST /api/auth/login`
Request JSON:
```json
{
  "email": "string",
  "password": "string"
}
```

Validation rules:
- `email`: email format
- `password`: non-empty

### 3.4 Login endpoint success response
If using JWT:
```json
{
  "accessToken": "jwt-string",
  "tokenType": "bearer",
  "user": {
    "id": "uuid-string",
    "email": "string",
    "firstName": "string",
    "secondName": "string"
  }
}
```

> Actual token strategy (JWT vs session cookies) should be decided by the backend design.

## 4) Password storage requirements

- Never store raw `password`.
- Store `password_hash` generated by a secure hashing algorithm (e.g., `bcrypt` or `argon2`).
- During login:
  - Fetch user by `email`
  - Verify password by comparing `password` with `password_hash`

## 5) Example SQLAlchemy model (illustrative)

```python
import enum
import uuid
from sqlalchemy import Column, String, Date, DateTime, Enum, func
from sqlalchemy.dialects.postgresql import UUID
from sqlalchemy.orm import declarative_base

Base = declarative_base()

class GenderEnum(str, enum.Enum):
    male = "male"
    female = "female"
    other = "other"

class User(Base):
    __tablename__ = "users"

    id = Column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)

    first_name = Column(String(100), nullable=False)
    second_name = Column(String(100), nullable=False)
    date_of_birth = Column(Date, nullable=False)
    address = Column(String(255), nullable=False)
    gender = Column(Enum(GenderEnum), nullable=False)

    email = Column(String(255), nullable=False, unique=True, index=True)
    password_hash = Column(String(255), nullable=False)

    created_at = Column(DateTime(timezone=True), server_default=func.now(), nullable=False)
    updated_at = Column(DateTime(timezone=True), server_default=func.now(), onupdate=func.now(), nullable=False)
```

## 6) How to use this mapping in your FastAPI endpoints

### Register flow
1. Accept JSON body matching the Register schema.
2. Validate gender/email/date/password.
3. Check if `email` already exists.
4. Hash password → store user in `users` table.
5. Return `userId` (and optionally issue token).

### Login flow
1. Accept JSON body matching the Login schema.
2. Find user by `email`.
3. Verify password.
4. On success, create token/session and return it.
